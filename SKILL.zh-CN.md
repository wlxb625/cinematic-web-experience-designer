# 电影化网页体验设计师

## 目标

生成一种真正融合布局、字体、图片、视频、3D、动效、交互和响应式适配的高级网页体验。

不要从产品需求直接跳到 React 组件。必须先确定视觉世界、唯一主角、页面分幕、媒体职责、动效逻辑、移动端重构方式和性能策略，最后才进入代码。

## 适用场景

适用于：

- 高端品牌官网和 Landing Page。
- AI 原生产品的品牌体验。
- 沉浸式首次引导和产品揭示。
- 互动作品集、工作室网站和叙事网站。
- 产品发布、年度回顾、记忆时间线和成长报告。
- 结合视频、图片、Shader、WebGL、3D 或滚动动画的网页。
- 用户提出“高级、震撼、电影感、沉浸式、丝滑、空间感、Awwwards 风格”等要求时。

不应把完整电影化方案用于普通后台、复杂数据表格、设置页面或高频操作界面。日常产品界面以清晰、稳定和安静为主，仅在关键时刻使用视觉高潮。

## 核心原则

高级感不是大量效果的集合，而是一个由统一概念和交互逻辑控制的视觉世界。

每个视觉效果至少应承担一种职责：

1. 引导注意力。
2. 表达层级和空间。
3. 说明状态变化。
4. 连接两个场景。
5. 揭示信息。
6. 强化产品含义。
7. 反馈用户操作。

只为“看起来高级”而存在的效果必须删除。

## 强制反模板规则

禁止默认生成常见 AI SaaS 模板：

- 蓝紫色科技渐变。
- 通用玻璃拟态。
- 全页面大圆角卡片。
- 无意义的渐变光球。
- 居中 Hero 下方三张功能卡片。
- 随机发光描边。
- 没有叙事职责的装饰图标。
- 所有元素分别执行 fade-up。
- 桌面布局在移动端简单纵向堆叠。
- 页面完成后再随意塞入素材。

最终页面不能像组件库示例或 AI 模板拼装。

## 体验类型判断

开始前必须判断当前界面属于哪一类：

### A. 品牌体验层

可以强调视觉冲击，例如官网、发布页、品牌活动和作品集。

### B. 关键产品时刻

选择性使用电影化体验，例如首次创建 AI、目标完成、年度回顾、知识图谱首次展开。

### C. 日常产品界面

优先保证阅读、效率和长期使用。只保留环境动效、状态反馈和关键转场，不能持续高刺激。

## 工作流程

必须按顺序完成以下阶段。第一至第六阶段没有确定前，不得进入实现。

### 阶段一：产品与情绪诊断

明确：

- 产品是什么。
- 用户进入页面要完成什么。
- 目标用户是谁。
- 第一感受应该是什么。
- 长期感受应该是什么。
- 不应该出现什么感受。
- 核心行动或转化目标。
- 属于 A、B、C 哪一种体验。
- 目标设备和性能假设。
- 无障碍和减少动态效果要求。

先输出简洁诊断。

### 阶段二：视觉世界定义

建立一个完整视觉世界：

- 核心隐喻。
- 唯一主视觉。
- 世界由什么材质或视觉物质构成。
- 空间层级。
- 光线行为。
- 动作动词。
- 交互含义。
- 色彩逻辑。
- 字体性格。
- 媒体质感。

动作动词示例：聚合、分裂、呼吸、结晶、消散、环绕、拉伸、显现、收束、重连。

不得只写“现代、科技、高端、未来感”，必须把形容词转化为可见行为。

### 阶段三：三个方向提案

在选定方案前，必须提出三种差异明显的方向：

1. 电影实拍或预渲染媒体方向。
2. 抽象空间或生成式视觉方向。
3. 编辑排版与动态字体方向。

每个方向都要比较：

- 情绪效果。
- 视觉冲击力。
- 开发复杂度。
- 素材制作成本。
- 性能风险。
- 移动端可行性。
- 是否适合长期使用。

最后给出明确选择和理由。

### 阶段四：页面分幕

页面必须以“场景”组织，而不是先写 Section 和组件。

每一幕必须定义：

- 目的。
- 情绪。
- 主视觉。
- 辅助内容。
- 空间构图。
- 媒体素材。
- 进入动效。
- 停留动效。
- 用户交互。
- 退出转场。
- 大致滚动长度或时间。
- 是否固定视口、Pin 长度、Scrub 强度和触发区间。
- 上一幕输入状态与下一幕输出状态。
- 移动端重新导演方式。
- 减少动态模式。

常见结构可为：

1. 进入。
2. 认识。
3. 互动。
4. 理解。
5. 高潮。
6. 收束或行动。

但不能机械套用，应根据产品重新安排节奏。

### 阶段五：媒体导演

每张图片、每段视频、每个 3D 物体、每种纹理或 Shader 都必须承担叙事职责。

每个素材需要定义：

- 类型。
- 叙事职责。
- 宽高比。
- 视觉焦点。
- 文字安全区。
- 桌面裁切方式。
- 移动端裁切或独立素材。
- 色彩分级。
- 循环方式。
- 加载策略。
- Poster 或占位。
- 性能降级。
- 减少动态版本。

禁止写“这里放一张高级图片”。

布局必须围绕素材焦点、人物朝向、画面留白和移动端裁切生成，而不是页面完成后再贴素材。

### 阶段六：动效与交互系统

所有视觉表现应由统一场景状态或主时间轴控制，不能让每个元素无原因独立运动。

可使用如下状态：

```ts
type ExperienceState =
  | "arrival"
  | "exploration"
  | "focus"
  | "reflection"
  | "conflict"
  | "resolution";
```

场景状态可以统一控制：

- 背景或 Shader 参数。
- 视频进度。
- 镜头位置。
- 字体对比度。
- 导航样式。
- 光标反馈。
- 粒子密度。
- 明确要求时的声音状态。
- 前景内容显隐。

滚动场景必须输出百分比时间轴，例如：

- 0%—15%：建立环境。
- 15%—35%：主体显现。
- 35%—55%：允许交互。
- 55%—75%：信息展开。
- 75%—90%：视觉高潮。
- 90%—100%：转入下一幕。

动效要表达因果关系，不得无意义持续漂浮。

#### 滚动编排是核心交互，不是附加装饰

当用户明确要求“下滑时有视觉变化、页面像影片一样展开、滚动过程震撼且流畅”时，必须把滚动视为**主时间轴控制器**，而不是在普通页面上添加几个进入动画。

每个重滚动场景都必须定义：

- `trigger`：场景从哪个容器触发。
- `start` / `end`：触发区间，例如 `top top` 到 `+=220%`。
- `desktop_scroll_vh` / `mobile_scroll_vh`：场景占用的滚动长度。
- `pin`：是否固定视口；不能默认每一幕都 Pin。
- `scrub`：滚动进度与动画时间轴的绑定强度。
- `progress`：归一化为 `0—1` 的唯一进度值。
- `input_state`：上一幕交给本幕的画面状态。
- `output_state`：本幕离开时交给下一幕的画面状态。
- `media_sync`：视频、序列帧、3D 镜头或 Shader 参数如何映射进度。
- `touch_behavior`：触屏设备如何重新编排。
- `reduced_motion`：减少动态模式如何保留信息因果关系。

推荐的固定场景结构：

```text
滚动容器：180vh—320vh
固定视口：100svh
滚动进度 p：0 → 1

p 0.00—0.10  预热素材并承接上一幕
p 0.10—0.28  建立环境与主视觉
p 0.28—0.52  主体发生核心变化
p 0.52—0.72  信息随变化被揭示
p 0.72—0.90  形成视觉高潮或认知结果
p 0.90—1.00  收束并把视觉状态交给下一幕
```

这不是固定模板，比例可以变化，但必须同时存在**进入、发展、结果、交接**，不能只在中间堆叠动画。

#### 场景交接连续性

相邻场景不能像两个独立网页突然切换。必须明确场景交接对象，例如：

- 上一幕的记忆核心缩小后，成为下一幕知识图谱的中心节点。
- 上一幕的背景光线扩散后，转化为下一幕的时间轴路径。
- 上一幕的一张图片放大到全屏，随后成为下一幕的视频首帧。
- 上一幕的关键词保持位置，在下一幕重组为标题。

每个场景规格必须包含：

```yaml
continuity:
  receives_from_previous: ""
  preserves: ""
  transforms: ""
  hands_to_next: ""
```

禁止通过纯黑闪屏、无意义淡出或突然换背景来掩盖缺少转场设计。

#### 可选滚动叙事模式

根据内容选择，不得为了炫技全部使用：

1. **渐进揭示**：内容随滚动逐层出现，适合说明关系和概念。
2. **固定变形**：视口固定，主视觉持续重组，适合核心产品揭示。
3. **镜头推进**：背景或 3D 摄像机进入不同空间层，适合沉浸叙事。
4. **视频进度映射**：滚动进度驱动预渲染视频 `currentTime`，适合电影化和复杂 3D。
5. **序列帧滚动**：Canvas 按滚动绘制图像序列，适合高精度产品拆解。
6. **横向旅程**：纵向滚动驱动横向场景，适合时间线；移动端通常改回纵向。
7. **状态重组**：同一批元素在不同状态间重排，适合记忆、观点和知识节点。

一页通常只需要 1—3 个重滚动场景，其余区域应恢复自然滚动，让用户有呼吸空间。

#### 技术实现规则

简单场景优先使用 CSS `position: sticky`、Transform、Opacity、Mask 和 CSS 变量。复杂场景才使用 GSAP ScrollTrigger。

GSAP 实现必须遵守：

```ts
useGSAP(() => {
  const timeline = gsap.timeline({
    scrollTrigger: {
      trigger: sceneRef.current,
      start: "top top",
      end: "+=220%",
      pin: true,
      scrub: 0.8,
      anticipatePin: 1,
      invalidateOnRefresh: true,
    },
  });

  timeline
    .fromTo("[data-subject]", { scale: 0.82, opacity: 0 }, { scale: 1, opacity: 1 })
    .to("[data-fragment]", { xPercent: 0, yPercent: 0, opacity: 1 }, "<35%")
    .to("[data-subject]", { scale: 0.56, xPercent: -24 }, ">")
    .fromTo("[data-next-title]", { opacity: 0 }, { opacity: 1 }, "<40%");
}, { scope: sceneRef });
```

实现要求：

- 使用 `gsap.context` 或 `useGSAP`，组件卸载时清理时间轴与 ScrollTrigger。
- 不得在每个滚动帧调用 React `setState`。
- 高频值使用 Ref、GSAP QuickSetter、CSS 变量或渲染循环读取。
- ScrollTrigger 数量应尽量按“每幕一个主时间轴”组织，不要为每个小元素创建独立 Trigger。
- `will-change` 只在活动场景使用，离开后移除。
- 页面字体和媒体加载完成后调用刷新，避免 Pin 高度跳动。
- 不得阻断浏览器原生滚动、反转滚轮方向或制造难以退出的滚动劫持。

#### 视频与序列帧同步

滚动驱动视频时，必须先确保视频元数据可用，并把归一化进度映射到可播放时长：

```ts
const targetTime = progress * Math.max(0, video.duration - 0.05);
video.currentTime = targetTime;
```

生产实现还需要：

- 使用适合随机 Seek 的编码和合理关键帧间隔。
- 首帧使用 Poster，进入触发区前预加载元数据或完整资源。
- 对高频 `currentTime` 更新做 requestAnimationFrame 合并。
- Safari/iOS 上准备预渲染短视频、图片序列或静态回退。
- 离屏后暂停视频并释放不再需要的序列帧缓存。

序列帧方案必须定义总帧数、预加载窗口、Canvas 尺寸、DPR 上限和缺帧处理。不能一次加载数百张原始大图阻塞首屏。

#### 平滑滚动使用边界

默认保留浏览器原生滚动。只有当项目确实需要统一触控板、滚轮和 WebGL 时间轴时，才引入 Lenis 等平滑滚动方案。

引入后必须：

- 与 ScrollTrigger 的刷新和 RAF 循环正确同步。
- 在 `prefers-reduced-motion` 下关闭平滑滚动。
- 保持键盘、锚点链接、浏览器查找和返回位置可用。
- 不使用过高阻尼让用户感觉页面拖拽或失控。

#### 滚动视觉效果验收

静态截图无法证明滚动体验完成。每个重滚动场景必须提交：

1. 场景 0%、25%、50%、75%、100% 五个进度截图。
2. 桌面端完整滚动录屏。
3. 手机端触摸滚动录屏或等价序列。
4. 快速滚动、反向滚动和停在中间状态的检查。
5. 低性能与减少动态模式录屏或截图。
6. 滚动结束后页面仍能继续自然浏览的验证。

验收时重点检查：

- 画面是否随滚动产生明确因果，而非单纯淡入。
- 停在任意进度时构图是否仍然成立。
- 快速滚动时是否闪烁、掉帧、跳位或错过关键信息。
- 反向滚动是否自然恢复，而不是播放错误状态。
- 场景进入和退出是否有连续交接。
- 用户是否能感知进度，但不会被强制困在动画里。

### 阶段七：响应式视觉导演

移动端是一套独立构图，不是缩小版桌面。

需要分别定义：

- 手机端视觉焦点。
- 手机端素材或裁切。
- 纵向节奏。
- 触摸交互。
- 导航方式。
- 字号和行宽。
- 固定场景长度。
- 动效简化。
- 加载优先级。
- 降级策略。

桌面端依赖 Hover 的交互，移动端必须改成触摸、滚动、点击或自动环境运动。

桌面使用实时 WebGL 时，移动端应评估替换为预渲染视频、序列帧、静态 Poster 或简化 Canvas。

### 阶段八：技术选型

始终选择能够实现目标的最简单技术，不能默认全部安装。

#### CSS 与浏览器原生能力

适合：简单位移、透明度、模糊、Sticky、遮罩、渐变和基础滚动关联效果。

#### Motion for React

适合：组件切换、共享布局动画、Hover、Tap、Focus、弹窗和产品级交互反馈。

#### GSAP 与 ScrollTrigger

适合：复杂滚动时间轴、固定场景、Scrub、文字、媒体和镜头的统一编排。

#### HTML Video

适合：电影实拍、AI 视频、预渲染 3D，以及不需要直接操控的复杂运动。

#### Canvas、WebGL 与 Shader

适合：生成式环境、粒子场、扭曲、自定义合成和逐帧渲染效果。

#### React Three Fiber 或 Three.js

只在主视觉确实需要真实 3D 操作、空间镜头、材质变化、产品配置或物体交互时使用。

### 阶段九：实现架构

推荐结构：

```text
experience/
  ExperienceShell
  SceneController
  scenes/
  media/
  motion/
  state/
  fallbacks/
ui/
  typography/
  controls/
  overlays/
```

必须分离：

- 视觉场景状态。
- 业务状态。
- 媒体加载状态。
- 减少动态模式。
- 设备能力模式。

避免在每一帧滚动中更新 React State。优先使用 Ref、动画库、requestAnimationFrame、CSS 变量或合适的外部状态管理。

### 阶段十：性能策略

每个复杂效果都必须有三级方案：

- Level A：高性能设备完整效果。
- Level B：普通设备简化动画或预渲染媒体。
- Level C：静态或减少动态版本。

必须明确：

- 首屏媒体预算。
- 延迟加载资源。
- 视频格式和 Poster。
- 图片格式与尺寸。
- 懒加载阈值。
- WebGL 能力检测。
- DPR 上限。
- 离屏暂停。
- 页面切换后的动画清理。
- 纹理和场景内存释放。

禁止首屏同时自动播放多段重视频。

### 阶段十一：可访问性

必须提供：

- `prefers-reduced-motion` 模式。
- 键盘导航。
- 清晰的 Focus 样式。
- 足够的文字对比度。
- 图片替代文本或等价内容。
- 声音控制。
- 不依赖动画也能获取核心信息。
- 核心交互不能只依赖鼠标位置。

视觉效果不能遮挡内容或阻止用户完成任务。

### 阶段十二：视觉验收

至少检查：

- 1440×1000 桌面。
- 1280×800 笔记本。
- 需要时检查 768×1024 平板。
- 390×844 手机。
- 375×812 手机。

必须输出截图；重动效页面还应输出短录屏。

检查：

- 每一幕是否只有一个清晰主角。
- 媒体是否真正融入构图。
- 标题与主视觉是否争夺注意力。
- 每个动画是否都有原因。
- 页面是否长时间持续高刺激。
- 移动端是否重新设计。
- 降级版本是否可用。
- 动态媒体上的文字是否始终清晰。
- 页面和场景转场是否连续。
- 重媒体完成前是否已有有效内容出现。

## 输出模式

### 概念模式

输出诊断、三个方向、选定世界和分幕草案，不写代码。

### 规格模式

输出完整场景、媒体、动效、响应式、技术和验收规格，不主动进入实现。

### 实现模式

只有在已有规格后才输出架构和代码。如果没有规格，应先补齐规格。

### 审计模式

审查已有网页或代码，识别：

- 模板化结构。
- 多个主视觉争夺注意力。
- 素材后贴。
- 动效缺少统一逻辑。
- 移动端缺少视觉导演。
- 性能风险。
- 可访问性问题。

按视觉收益和实现成本排序修改优先级。

## Codex 执行协议

把任务交给 Codex 或其他编码代理时：

1. 先审查代码库并列出约束。
2. 未经要求不得修改业务逻辑和数据契约。
3. 一次只实现一个场景或一个系统层。
4. 每阶段完成后生成截图。
5. 对照场景规格检查实现偏差。
6. 删除没有改善层级和含义的效果。
7. 验证手机端和减少动态模式后再继续。
8. 汇报修改文件、技术取舍、剩余风险和下一步验证内容。

禁止一次性要求“把整个网站改得高级”。

## 完成标准

只有满足以下条件才算完成：

- 体验拥有明确命名的视觉概念。
- 每一幕都有清晰的唯一主视觉。
- 所有媒体都有叙事职责。
- 动效由统一状态或主时间轴控制。
- 重滚动场景拥有明确的 Trigger、Pin、Scrub、进度映射和场景交接。
- 已通过完整滚动录屏，而不只是静态截图。
- 桌面和移动端均经过独立设计。
- 存在减少动态和低性能降级。
- 已在目标尺寸完成视觉验收。
- 没有电影化效果时，核心业务仍可正常完成。
- 页面不再像通用 AI 模板。
