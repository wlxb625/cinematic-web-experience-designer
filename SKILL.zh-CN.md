# 电影化网页体验设计师

## 强制执行顺序

不能从需求直接跳到组件。必须按照以下顺序工作：

1. 诊断产品、用户、核心任务、情绪目标和页面类型。
2. 条件允许时检查现有网站、代码、截图、媒体资源和技术约束。
3. 当前任务需要高端视觉或当前案例时，先完成外部参考调研。
4. 填写 `templates/reference-research.yaml`，筛选可迁移原则。
5. 阅读 `references/core-workflow.zh-CN.md`，执行完整的视觉世界、页面分幕、媒体导演、滚动时间轴、移动端重构、性能、无障碍、实现与验收流程。
6. 建立一个统一视觉系统，再逐幕实现。

参考调研是设计输入，不是照搬许可。

## 强制参考调研

出现以下任一情况，并且当前环境具备网页搜索或浏览器工具时，必须在确定视觉世界前调研：

- 用户要求高级、电影感、沉浸式、强动效、Awwwards 风格、Bento 风格或明显区别于模板的 UI。
- 用户提供网站、截图、Moodboard、Pinterest 画板或指定参考来源。
- 推荐结果依赖当前组件库、浏览器能力、动效实现方式或设计趋势。
- 页面包含动态 Hero、下滑联动、复杂背景、特殊导航、模块化叙事或空间构图。

只有在修改已批准设计中的局部内容，或用户已经提供完整且当前有效的参考集时，才可以跳过外部调研。

没有真正打开的网站，不得声称已经查看。网页访问不可用、网站要求登录或被阻止时，改用用户提供的链接内容、截图、录屏、导出文件和项目资料，并明确标注尚未验证的部分。

## 指定参考来源的分工

每个网站承担不同职责，不能把它们混成无规则的组件拼贴。

### React Bits：动效模式参考

使用 React Bits 研究动态文字、背景、光标、卡片、滚动效果和交互组件。

每个候选效果都要检查：

- 实际运行行为和节奏，不能只看截图。
- 可配置参数和依赖。
- React 兼容性与样式假设。
- 手机端和触摸行为。
- `prefers-reduced-motion`，或需要补充的减少动态方案。
- 性能成本、卸载清理和同时运动的图层数量。

每个视觉方向通常只选择 1—3 个真正强化场景概念的模式。不能因为组件存在就叠加使用。

### 21st.dev：经过审查的实现候选库

使用 21st.dev 搜索 React/Tailwind 组件、营销区块、Hero、网格、导航、画廊、AI 界面、Shader 和相关 UI。

每一项都视为需要审查的社区代码。采用前必须确认：

- 作者、来源以及授权或访问要求。
- 依赖、版本以及与当前项目的冲突。
- 无障碍、键盘、响应式和语义结构。
- 样式约定与现有 Design Tokens 是否兼容。
- 是否可以提取必要原理，而不是整块照搬。

采用后必须改造成项目自己的架构和视觉语言。不得把互不相关的组件组合成视觉拼盘。

### MotionSites：整体网站构图和节奏参考

使用 MotionSites 研究系统级问题：

- Hero 到正文的节奏。
- 主视觉焦点和信息层级。
- 图片、视频和界面的融合方式。
- 区块或场景之间的转场。
- 转化路径和品牌一致性。
- 视觉高潮与安静阅读区域之间的平衡。

只能提取系统原则，不得复制具体布局、文案、素材、品牌或标志性转场。

### BentoGrids：模块层级和响应式重组参考

使用 BentoGrids 研究卡片比例、不对称构图、视觉锚点、混合媒体、信息密度和手机端重组。

只有内容确实适合模块比较、多信号展示或结构化叙事时才使用 Bento。不能默认所有页面都使用 Bento。手机端必须按优先级重新编排，不能机械按照桌面顺序纵向堆叠。

### Pinterest：宽范围美术方向与 Moodboard

使用 Pinterest 扩展字体、摄影、编辑节奏、色彩、材质、光线、空间构图和整体美术方向。

搜索词必须结合具体产品和情绪，不能只搜索“高级网页”。Pinterest 可能要求 JavaScript 或登录；访问受阻时，使用用户提供的画板、截图或导出资料，不得虚构观察结果。

Pinterest 图片只用于情绪和美术方向，不自动等同于可复用素材或实现规格。

## 调研流程

1. 根据产品、用户、情绪、内容类型、视觉隐喻和目标交互生成搜索词。
2. 从至少两类来源中选择 2—5 个强参考。
3. 动效参考必须查看真实交互或录屏；静态截图不能证明节奏和滚动行为。
4. 将每个实际查看的来源写入 `templates/reference-research.yaml`。
5. 分别提取构图、字体、媒体、动效、交互、响应式、无障碍和性能原则。
6. 记录哪些内容不能复制。
7. 把每个采纳原则映射到具体场景、组件和产品目的。
8. 没有明确产品职责或不适合长期使用的参考必须删除。

## 组件采用门槛

React Bits 或 21st.dev 中的组件和效果，只有同时满足以下条件才能进入实现：

- 支持明确的产品、叙事、层级或交互目的。
- 已检查来源和复用条件。
- 已记录依赖、版本和框架兼容性。
- 改造后能进入现有 Design Tokens 和项目架构。
- 已定义键盘、触摸、手机端、减少动态和低性能行为。
- 卸载清理、路由切换和高频动画更新安全。
- 已优先考虑更简单的 CSS、浏览器原生能力或项目现有方案。

只具有视觉吸引力、但没有信息或交互职责的效果必须放弃。

## 原创性规则

禁止复制参考网站的具体布局、比例、文案、品牌、媒体、插画、代码结构或标志性转场，也不能通过拼接多个参考区块来复刻一个可识别页面。

每个采用原则至少要改变以下三个维度：

- 内容层级。
- 空间构图。
- 字体系统。
- 色彩和材质。
- 媒体主体和裁切。
- 动效原因和节奏。
- 交互含义。
- 响应式行为。

最终设计必须能从产品自身概念解释，而不是依赖参考网站的身份。

## 下滑动效要求

当用户强调下滑过程中的视觉体验时，滚动必须成为主时间轴，不能只给普通页面增加几个淡入动画。具体执行 `references/core-workflow.zh-CN.md` 中的 Pin、Scrub、统一进度、场景交接、视频或序列帧同步、手机端、减少动态和录屏验收规则。

没有关键进度截图以及桌面端、手机端录屏，不得声明动效已经完成。

## 编码前必须输出

1. 产品与情绪诊断。
2. 完整参考调研记录。
3. 三个差异明显的视觉方向。
4. 选定的视觉世界及理由。
5. 页面分幕和场景交接链。
6. 媒体规划。
7. 动效与滚动时间轴。
8. 桌面端和手机端视觉导演。
9. 技术与组件采用决策。
10. 性能、无障碍与验收计划。

代码是最后阶段，不是第一阶段。
